9) CONTAINERS ::
-------------------
     ** containers are light weight, portable units that package an application and its dependencies like (libraries and binaries , configuration files)into a single isolated environment.
     ** traditional VM's , containers share the host system kernel but remaining isolated each other.
     ** containers are typically created using containerization platforms like ""Docker or Kubernetes"".
     ** A container image is a ""standalone"" , executable package that includes everything needed to run a piece of software.

Why containers are popular??
-------------------------------
1. Portability::
-----------------
         * Containers encapsulate applications and their dependencies, making them platform-agnostic.
         * Developers can build containers on their local machine, and they’ll run consistently in any environment—whether on-premises, in the cloud, or in hybrid setups.

2. Isolation::
---------------
         *Each container runs in its own isolated environment, ensuring that dependencies and configurations don’t conflict with others.
         *This is particularly useful in DevOps, where multiple applications or versions often run simultaneously on shared infrastructure.

3. Lightweight and Efficient:
-------------------------------
         * Containers are much smaller than virtual machines (VMs) because they share the host OS kernel rather than bundling a full operating system.
         * This reduces resource overhead and allows faster startup times.

4. Faster Development and Deployment:
------------------------------------
         * Containers enable developers to package applications with all dependencies into a single unit, simplifying the build and deployment process.
         * DevOps teams can integrate containers into CI/CD pipelines for automated building, testing, and deployment.

5. Scalability:
---------------
          * Containers can be easily replicated to handle increased loads, making them ideal for scaling applications dynamically.
          * Container orchestration tools like Kubernetes and Docker Swarm automate scaling, load balancing, and failover.

6. Consistency Across Environments:
------------------------------------
          * Containers eliminate the "works on my machine" problem by ensuring the same runtime environment across development, testing, staging, and production.

7. DevOps Tool Integration:
---------------------------
          * Containers integrate seamlessly with DevOps tools and practices such as:
                 ------ > CI/CD tools: Jenkins, GitLab CI/CD, CircleCI.
                 ------ > Monitoring tools: Prometheus, Grafana, Datadog.
                 ------ > Version control: Docker images can be versioned and stored in container registries.

8. Microservices-Friendly:
---------------------------
          * Containers are an ideal fit for microservices architectures, where applications are divided into smaller, independently deployable services.
          * Each microservice can be deployed in its own container, enabling independent development and scaling.

9. Improved Resource Utilization:
----------------------------------
          * Containers use fewer resources than VMs because they share the OS kernel. This allows running more containers on the same hardware compared to VMs.

10. Rapid Start-Up Times:
-------------------------
          * Containers start in seconds (or less), enabling faster deployments and reducing downtime.

11. Easier Rollbacks:
----------------------
          * Containers can be easily versioned, allowing DevOps teams to roll back to a previous version if an update causes issues.

DIFFERENCE BETWEEN VIRTUAL MACHINE AND CONTAINER::
-------------------------------------------------------
  Aspect	                            Virtual Machines (VMs)	                                   Containers
----------------                           --------------------------                                     --------------
Abstraction Level	         Hardware-level virtualization.	                             OS-level virtualization.
Components	                 Includes a full OS, libraries, and applications.	     Includes only the app and its dependencies; shares host OS kernel.
Hypervisor/Engine	         Requires a hypervisor (e.g., VMware, Hyper-V, KVM).	     Uses a container runtime (e.g., Docker, container).
Size	                         Larger (e.g., gigabytes) due to the full OS.	            Smaller (e.g., megabytes) since only the app and its dependencies are packaged.
Startup Time	                 Slower (minutes) as the full OS must boot.	            Faster (seconds or less) since it uses the host OS kernel.
Resource Overhead	         High due to having its own OS and kernel.	            Lightweight; shares the host OS kernel, reducing overhead.
Isolation	                 Stronger, as each VM has its own OS and kernel.	    Weaker, as containers share the host OS kernel.
Security	                 More secure; vulnerabilities are contained within the VM.  Less secure; exploits may affect the host or other containers.
Portability	                 Less portable due to larger size and dependency on hypervisor.	Highly portable across environments (cloud, on-premises, local).
Performance	                 Slower due to OS overhead and emulated hardware.	     Faster as they directly leverage the host OS.
Scaling	                         Resource-intensive and slower to scale.	             Quick and efficient scaling.
Use Cases	                 - Legacy applications                                                 - Microservices
                                 - Running multiple OS types                                           - DevOps workflows (CI/CD)     
                                 - High-security needs.	                                               - Modern cloud-native apps.
Examples	                 VMware, Hyper-V, KVM, VirtualBox.	                      Docker, Kubernetes, Podman, containerd.

AGENTS::
--------
     ***An Agent in Azure DevOps is a software component that executes build and deployment tasks in a pipeline on a physical or virtual machine.        

Agents Work in DevOps::
-------------------------
         1) Agent Connection:
         --------------------
                      ** Agents are connected to the DevOps tool (e.g., Jenkins, Azure DevOps, GitLab CI, GitHub Actions) and listen for jobs                                             
                                               assigned by the server or controller.
         2) Environment Setup:
         ---------------------
                      ** Each agent operates in an environment (e.g., a virtual machine, container, or bare-metal server) with pre-installed  
                                           tools and dependencies necessary for the job.
         3) Execution of Jobs:
         ---------------------
                      ** When a job is triggered in the pipeline, the server assigns it to an available agent.
                      ** The agent pulls the required code, dependencies, or configurations and executes the job.
         4) Reporting Results:
        ----------------------
                      ** The agent sends the output and status of the job (success or failure) back to the server.


Use Cases of Agents in DevOps::
-------------------------------
          1) Building and Testing Code:
          -----------------------------
                 ** Agents compile source code, run automated tests, and generate build artifacts.
          2) Deploying Applications:
          --------------------------
                 ** Agents handle deployment tasks to production, staging, or testing environments.
          3) Running Automation Scripts:
           ------------------------------
                 ** Execute infrastructure-as-code tools like Terraform or Ansible.
          4) Container Management:
          -------------------------
                  ** Agents can build and push Docker images or manage Kubernetes clusters.
          5) Static Code Analysis:
          --------------------------
                  ** Perform linting or code quality checks using tools like SonarQube.
         6) Parallel Execution:
          ----------------------
                   ** Run jobs in parallel across multiple agents to reduce pipeline execution time.

PPELINE TRIGGERS::
------------------
    ***  we can trigger a pipelines based on repository activity , scheduled runs ,manual interventions . 

1) Continuous Integration trigger ::
------------------------------------
   * CI triggers automatically run a pipeline when changes are made to the repository .
   * Every time code is pushing to the repository , the pipeline is triggered to ensure the code remains integrated and free of breaking changes.

trigger:
 branches:
  include:
    -main 
    -dev       ###  list of branches to be included 
  exclude:
    -feature/*    ### list of branches to be excluded 

includes::
----------
   * specifies which branch will trigger the pipeline when changes are pushed

excludes::
----------
   * Specifies the branch that are excluded from the CI trigger.

2) Scheduled Trigger::
----------------------
   * Scheduled triggers are allow pipeline to run at specific times . regardless of whether there have been code changes .
   * this is useful for daily or nightly builds batch processing or running tests on a regular basis.

   -cron:"0 2 */4**"               ### cron expression to define the schedule (runs at 2am daily)
     displayName: Daily build 
        branches:
         include:
          -main

CRON ----- > A cron expression defining when the pipeline should run. in the above example , the pipeline line runs at daily 2am.

3) PIPELINE TRIGGERS::
----------------------
  ** A pipeline Triggers runs when another pipeline runs completely . this is useful for scenarios where you have multiple pipelines and want them to run sequentially 

    trigger:
     none                   ### Disable CI pipeline for this pipeline
   resources:
   pipelines:
    -pipeline:'pipelineA'    ### the pipeline is to depend on.
   source:'myproject-projectA'      ### the source pipeline name.
  trigger:
     branch:
      include:	
       -main

pipeline-----> the name of the resource pipeline you want to trigger the current pipeline.
source ------> the name of the other pipeline to trigger from.
branch ------> Filters the branch on which the trigger occurs.

4) PULL REQUEST TRIGGER::
-------------------------
    * PR triggers start a pipeline when a pull request is raised in the specified branch or set of branches. 
    * this helps run builds and tests as soon as code is submitted for review.

pr:
 branches:
  include:
    -main
    -dev

branch.includes---->specifies the branches where a pull request will trigger the pipeline.

5) PATH TRIGGER::
-----------------
    ** path trigger allows you to run a pipeline when changes are made in specific paths or directories with the repository.
     *this is useful for monorepos.

triggers:
   paths:
    include:
      -src/*
    exclude:
      -docs/*

include----> specifies the paths that will trigger the pipeline when changes are detected
exclude ---> specifies paths that should be ignored.

The pipeline will only trigger if there are changes in the src/ directory , and it will ignore changes made in the docs/ directory.


JOBS ::
-------
        ** In DevOps, jobs represent a fundamental unit of work within automated pipelines used to build, test, deploy, or manage software and infrastructure. 
                    Jobs consist of a series of steps that accomplish specific tasks, such as compiling code, running tests, or deploying applications. 

1) MULTI AGENT JOBS::
----------------------
        *** Multi-agent jobs in DevOps pipelines are used to distribute tasks across multiple agents, enabling parallel execution and improving efficiency.
                                                       (OR)
         ** A multi -agent jobs runs the same task on multiple agents in parallel. this is useful for scenarios where you want to 
                           speed up the build or test process by distributing work across multiple agents

Key Concepts of Multi-Agent Jobs::
----------------------------------
       1) Parallelism:
       ---------------
              * Tasks within a pipeline can run simultaneously on multiple agents.
              * Each agent operates independently with its own resources.
       2) Agent Pools:
       ---------------
              * Jobs can target specific agent pools for specialized tasks (e.g., Linux vs. Windows agents).
       3) Use Cases:
       -------------
              * Testing across multiple environments (e.g., different operating systems, browsers).
              * Building artifacts for multiple platforms (e.g., ARM, x86).
              * Parallel deployments to multiple regions.
       4) Dependencies:
       ----------------
              * Outputs of one job can be passed as inputs to another job.
jobs:
 -job:Runtests
  displayName:"Run Test on Multiple Agents"
  pool:
   name:"Azure Pipelines"
   demands:
    -agent.os-equals"Windows"
  strategy:
    parallel:3    ## run the job on 3 parallel agents
   steps:
    -task:UseDotNet@2
     inputs:
      packageType:"sdk"
      version:"6.x"
      installationPath:$(Agent.ToolsDirectory)/dotnet
   -script:|
        dotnet restore
        dornet test--filter FullyQualifiedname~UnitTests
       displayName:"Run Unit Tests"


---> The Parallel: 3 settings tells Azure DevOps to run the RunTest Job on 3 different agents in parallel

2) MULTI-CONFIGURATION JOBS::
-----------------------------
         *** Multi-Configuration Jobs in DevOps (also known as Matrix Jobs) allow pipelines to run a job or stage multiple times with varying configurations. 

Key Benefits of Multi-Configuration Jobs::
--------------------------------------------
         Efficiency:
         -----------
             * Parallel execution reduces the overall pipeline time.
         Comprehensive Coverage:
         -----------------------
             * Ensures the application works across all supported environments.
         Scalability:
        --------------
             * Easily scales to add more configurations.
         Dynamic Configuration:
         -----------------------
             * Can adjust configurations dynamically for specific use cases.

trigger:
  - main
jobs:
  - job: Test
    displayName: "Matrix Testing"
    strategy:
      matrix:
        Linux:
          vmImage: 'ubuntu-latest'
          NODE_VERSION: '14'
        macOS:
          vmImage: 'macos-latest'
          NODE_VERSION: '16'
        Windows:
          vmImage: 'windows-latest'
          NODE_VERSION: '14'
    pool:
      vmImage: ${{ matrix.vmImage }}
    steps:
      - task: UseNode@2
        inputs:
          version: ${{ matrix.NODE_VERSION }}
      - script: |
          echo "Running tests on ${{ matrix.vmImage }} with Node.js ${{ matrix.NODE_VERSION }}"
          npm install
          npm test

Best Practices for Multi-Configuration Jobs::
---------------------------------------------
           Minimize Overlap:
           -----   ----  ----
                 * Avoid redundant configurations to save time and resources.
           Optimize Resources:
           -------------------
                 * Use dynamic agents and adjust concurrency limits to prevent resource bottlenecks.
           Aggregate Results:
           ------------------
                 * Ensure proper reporting and aggregation of job outputs for easier analysis.
           Fail Fast:
           ----------
                 * Configure pipelines to stop further runs if critical configurations fail.
           Environment Isolation:
           ----------------------
                 * Use containers or VMs to ensure each configuration runs in an isolated environment.
